# Agent 技术面试题（整理稿）

> 来源：录音《汉堡王(深圳留芳路店)》讨论要点 + 技术思路补充。
> 录音核心框架（用户复盘）：agent 分 **闲聊 / 任务 / coding** 三类；**coding 滑入任务**；任务实现方式很多；重点讨论 **闲聊** 与三类共用的**记忆/上下文机制**（实体抽取、压缩、检索、幻觉治理、KM）。
> 用途：面试官出题 / 候选人准备。每题含「考察点」与「参考思路」（"你的思路"部分）。

---

## 〇、背景速记（供出题人把握考点）

- **分类**：闲聊（chitchat）、任务（task）、coding（归入任务）。
- **共用底座**：后台统一 State、上下文压缩、实体抽取与分组、记忆检索、幻觉治理。
- **重点**：闲聊 agent 的记忆策略与"实体放顶层导致幻觉放大"的风险。

---

## 一、Agent 分类与意图路由

**Q1. 如何把用户请求分为「闲聊 / 任务 / coding」三类？并说明 coding 为什么可以归入任务。**
- 考察点：意图分类、路由设计、对 agent 本质的理解。
- 参考思路：
  - 用轻量分类器（小模型/规则+LLM 兜底）输出 `{type, confidence}`，低置信回落到 LLM 判断。
  - coding 本质是"通过工具/执行产生产物并验证"的任务：写代码→运行→看报错→修正，构成**执行+状态闭环**，与通用任务共享同一套 task loop，因此可归并。
  - 区分信号：是否改变外部状态、是否需工具调用、是否有可验证产物。

**Q2. "coding 滑到任务去"——在架构上，coding agent 与 task agent 如何复用同一套执行/状态框架？**
- 考察点：框架抽象能力、复用设计。
- 参考思路：抽象出 `TaskExecutor`（规划→工具调用→状态更新→校验），coding 只是把"工具"换成编译器/解释器/测试，把"产物"换成代码与运行结果；State 结构通用。

**Q3. 闲聊 agent 与任务 agent 在目标、工具使用、记忆策略上有何本质区别？**
- 考察点：是否真正理解"闲聊"的特殊性（避免把闲聊也搞成重型 pipeline）。
- 参考思路：
  - 目标：闲聊=陪伴/情绪/人设一致；任务=完成可验证结果。
  - 工具：闲聊默认**不调外部工具**，避免"为了聊天去查数据库"的过度工程。
  - 记忆：闲聊记"用户偏好/人设上下文"，任务记"步骤/产物/状态"。

## 二、任务执行与状态

**Q4. 任务执行有"规范做法"，如何设计任务的后台统一 State？为什么说"不能丢"？**
- 考察点：状态持久化、可靠性工程。
- 参考思路：State 作为独立持久化对象（task_id、步骤、产物、status），落库（DB/Redis/文件）而非只存上下文；用写前日志/checkpoint 保证"不能丢"；上下文只是视图，State 才是真相源。

**Q5. 任务实现方式很多（ReAct / Plan-Execute / 多 agent / 工作流），各自的适用边界？**
- 考察点：方案权衡，而非只会一种。
- 参考思路：
  - ReAct：灵活、适合探索型。
  - Plan-Execute：可控、适合复杂长任务。
  - 多 agent：适合职责解耦，但有协调开销。
  - 工作流/DAG：确定性高、适合可固化流程。

**Q6. 如何保证长任务过程中关键信息不被上下文压缩掉？**
- 考察点：压缩与保真的平衡。
- 参考思路：分层——最近若干轮原文保留；更早的滚动摘要；**已 commit 的事实/实体**抽成结构化记忆，不参与摘要，避免被"压掉"。

## 三、上下文与记忆管理（重点）

**Q7. 上下文窗口有限，如何在不丢失关键信息的前提下压缩？描述你的分层策略。**
- 考察点：上下文工程核心能力。
- 参考思路：三层——① 工作区（最近 N 轮原文）；② 滚动摘要（旧轮压缩）；③ 结构化长期记忆（实体/事实，按需检索注入）。压缩时保留"决策点"与"未决项"。

**Q8. 如何从对话中抽取"重要实体/话题"？如何归类形成实体组？**
- 考察点：信息抽取 + 知识组织。
- 参考思路：LLM 带 schema 抽取（实体、类型、置信度），去重合并；用类型体系（人/组织/概念/任务…）做"归类/归属"；同主题实体聚成"实体组（记忆组）"持久化。

**Q9. 新对话如何检索历史记忆？如何"快速找到最相关的东西"？**
- 考察点：检索架构。
- 参考思路：混合检索（稠密向量 + BM25/关键词）+ 重排（reranker）；结合**时效性 + 相关性**打分；用查询改写/扩展提升召回；命中后做证据片段回传。

**Q10. 实体组之间如何建立关系（图谱化）？"边"从哪来？**
- 考察点：知识图谱思维。
- 参考思路：实体为节点，关系（边）由 LLM 在抽取时一并产出（主语-谓词-宾语）；边支持多跳推理；可定期对齐去歧义。

**Q11. 短期记忆 / 长期记忆 / 用户画像如何分层？**
- 考察点：记忆系统建模。
- 参考思路：短期=当前会话；长期=跨会话 episodic/semantic 记忆；用户画像=稳定偏好与事实。三者写入策略与 TTL 不同。

## 四、幻觉与可靠性（重点）

**Q12. 为什么"把实体放在顶层/一级"会让幻觉变大？**
- 考察点：对幻觉传播机制的深层理解。
- 参考思路：未经验证的抽取实体若直接注入系统提示/顶层上下文，模型会将其当作**既定事实**采纳，错误沿生成链放大；顶层位置=最高置信注入。

**Q13. 实体本身可能"有错/有偏"，抽取阶段如何校验（"不能随意抽随意改"）？**
- 考察点：数据质量治理。
- 参考思路：抽取即校验——schema 校验、与已有记忆矛盾检测、来源可追溯（provenance）、低置信标灰而非直接入库；写入走 review/合并流程。

**Q14. 幻觉已产生时，如何"找证据来支撑、消除幻觉"？**
- 考察点： grounding / 事实核查。
- 参考思路：RAG 回源、要求引用出处、self-consistency 多采样、工具验证（查库/跑代码）、矛盾检测与回滚。

**Q15. 实体应放在哪一层？除了顶层还有哪些位置可选？**
- 考察点：记忆分层与注入策略。
- 参考思路：放"检索层"而非"事实层"——平时不注入，仅在被检索命中且经相关性过滤后再进入上下文；标注 provenance 与置信度，让模型知其"待证"。

## 五、闲聊（chitchat）专题

**Q16. 闲聊 agent 的记忆应该记什么、不记什么？与任务记忆有何不同？**
- 考察点：闲聊场景的记忆边界。
- 参考思路：记用户偏好、人设上下文、重要个人事实；**不记**瞬时寒暄、易变情绪。与任务记忆隔离，避免任务状态污染闲聊人设。

**Q17. 如何让闲聊 agent 保持人设一致、又不滥用工具/不胡乱调外部系统？**
- 考察点：人设一致性 + 克制调用。
- 参考思路：人设写在只读 persona 层；默认禁工具，仅在用户明确请求外部动作时开放；用少量示例约束"何时该聊天、何时该行动"。

**Q18. 闲聊中如何自然地运用 KM（知识库）而不显生硬？**
- 考察点：知识注入的自然度。
- 参考思路：按需检索、以"分享/补充"口吻自然带出，而非堆砌；命中即带引用，未命中不硬塞。

## 六、系统设计与开放题

**Q19. 端到端设计一个 agent 记忆系统**（抽取 → 存储 → 检索 → 压缩 → 治理）。
- 考察点：系统思维完整性。
- 参考思路：抽取（LLM+schema+校验）→ 存储（向量+图+KV，带 provenance/置信度）→ 检索（混合+重排）→ 压缩（分层摘要+保留事实）→ 治理（矛盾检测、TTL、人工可纠）。

**Q20. 长上下文模型流行后，检索增强还有必要吗？**
- 考察点：技术趋势判断。
- 参考思路：仍有必要——长上下文贵、且缺乏 provenance；RAG 提供可溯源与跨超长周期的记忆，二者互补而非替代。

**Q21. 如何评测一个 agent 的记忆/闲聊质量？**
- 考察点：评测体系。
- 参考思路：记忆召回率（重要事实 recall@k）、幻觉率、人设一致性（LLM-judge）、闲聊场景的"工具滥用率"、用户满意度。
