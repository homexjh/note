0.一句话总览

Context Parallel 是为长上下文推理服务的并行方式：prefill 阶段重点降低首 token 延迟 TTFT，decode 阶段重点减少 KV cache 占用、提高 batch size 和吞吐。

最通俗的理解：
Prefill 像“读题”：一次性读完整个 prompt。
Decode 像“答题”：一个 token 一个 token 往外生成。
PCP 是多人一起读长题。
DCP 是多人分摊存储超长历史笔记，每次答题时再合起来用。

1.关键符号和概念

符号 / 术语 含义 通俗解释
T 上下文 token 数 / 序列长度 模型当前要记住多少个 token。例如 128k 表示 128,000 个 token。
H 在 H * T 中表示 KV heads 数量 有多少组 K/V 头。KV cache 大小大致和 H × T 成正比。
H^(l) 第 l 层 hidden states 注意这里的 H 不是 KV heads，而是每个 token 的隐藏表示矩阵。
Q Query 当前 token “想找什么信息”。
K Key 每个历史 token “能被怎样匹配”。
V Value 真正被加权汇总的信息内容。
KV cache 缓存历史 token 的 K/V decode 时不用重复计算所有历史 token 的 K/V。
TTFT Time To First Token 从请求进来到第一个输出 token 出现的时间。
KV cache 粗略大小 ∝ 层数 × T × KV heads 数量 × head_dim × 2(K 和 V) × dtype_size

vLLM 文档中提到，对于有 H 个 KV heads、上下文长度为 T 的请求，需要在 KV cache 中存储 H * T 级别的 K/V 张量。

2.Q/K/V 怎么来的？

Q/K/V 不是输入文本里自带的，而是模型用当前 token 的 hidden state 乘以三组训练好的权重矩阵得到的。

Q = H × W_Q
K = H × W_K
V = H × W_V

对单个 token 来说：

q_i = h_i × W_Q
k_i = h_i × W_K
v_i = h_i × W_V

小例子

假设输入是：

我 爱 北京

token 化后：

token0 = 我 token1 = 爱 token2 = 北京

每个 token 先变成 embedding / hidden state：

我 → h0 爱 → h1 北京 → h2

在某一层里，模型用三组矩阵把 hidden state 投影成 Q/K/V：

h0 → q0, k0, v0 h1 → q1, k1, v1 h2 → q2, k2, v2
直观理解：
Query 像“问题”，Key 像“索引”，Value 像“答案内容”。
Attention 就是：当前 token 的 Query 去匹配所有可见 token 的 Key，然后按匹配程度加权汇总 Value。

3.第一层输出后得到什么？第二层又怎么操作？

Transformer 不是只算一次 Q/K/V，而是每一层都重新计算自己这一层的 Q/K/V。

输入 token ↓ Embedding ↓ H^(0) 第 1 层： H^(0) → Q^(1), K^(1), V^(1) → Attention → MLP / FFN → H^(1) 第 2 层： H^(1) → Q^(2), K^(2), V^(2) → Attention → MLP / FFN → H^(2) 第 3 层： H^(2) → Q^(3), K^(3), V^(3) → Attention → MLP / FFN → H^(3) ... 最后一层 H^(L) ↓ LM Head ↓ Logits ↓ 采样 / 贪心选择下一个 token
重点：第一层输出的是新的 hidden states H^(1)，不是最终文字。第二层拿 H^(1) 当输入，用第二层自己的 W_Q^(2), W_K^(2), W_V^(2) 重新算 Q/K/V。

4.Prefill 与 Decode

Prefill 读 prompt

用户的 prompt 已经全部给定，模型可以并行处理很多 token。

输出：prompt 的 KV cache，以及用于生成第一个 token 的最终 hidden state / logits。

主要影响：TTFT。

Decode 生成答案

自回归生成，每一步依赖上一步刚生成的 token。

每一步通常只有很少的新 query token，但要 attend 大量历史 KV。

主要影响：生成速度 tokens/s、吞吐、KV cache 显存。

为什么 prefill 可以并行，而 decode 不行？

prefill 的 token 全部已经存在，比如 prompt 是 token0 ... token127999，模型可以并行算这些 token 的表示，但用 causal mask 保证每个 token 不能看到未来。

decode 的第 10 个输出 token 必须等第 9 个输出 token 先生成出来，所以天然是串行的。

5.Context Parallel 要解决什么问题？

vLLM 文档把 context parallel 分成两个方向：

方向 阶段 目标
PCP / Prefill Context Parallel Prefill 长 prompt 的 prefill 太慢，降低 TTFT。
DCP / Decode Context Parallel Decode KV cache 太大，分摊显存，提高 batch size 和吞吐。
核心差异：
PCP 主要分摊“长 prompt 的计算”。
DCP 主要分摊“长上下文历史 KV cache 的存储和 attention 计算”。

6.PCP：Partial Query, Full Key/Value

vLLM 文档中说，prefill 时如果长请求有 T 个新 token、N 张 GPU，可以把请求切成 N 个 chunk，每张 GPU 计算一块 Q/K/V。如果请求长度“中等长”，单卡还能放下完整 K/V，那么可以 gather 所有 GPU 的 K/V，让每张 GPU 只计算自己那部分 query 对应的 attention output。

例子：128k prompt，4 张 GPU

GPU0: token 0 ~ 32k GPU1: token 32k ~ 64k GPU2: token 64k ~ 96k GPU3: token 96k ~ 128k

每一层内部流程

  1. 每张 GPU 拿自己那块 hidden states。
  2. 每张 GPU 本地同时计算自己那块的 Q/K/V。
  3. 只聚合 K/V，让每张 GPU 拿到完整 K/V。
  4. Q 不聚合，每张 GPU 保留自己的 Q。
  5. 每张 GPU 用自己的 Q attend 完整 K/V，得到自己那块 token 的 attention output。
  6. 每张 GPU 仍然保存自己那块输出 hidden states，进入下一层。
第 l 层： GPU0: H0^(l-1) → Q0^(l), K0^(l), V0^(l) GPU1: H1^(l-1) → Q1^(l), K1^(l), V1^(l) GPU2: H2^(l-1) → Q2^(l), K2^(l), V2^(l) GPU3: H3^(l-1) → Q3^(l), K3^(l), V3^(l) 通信聚合 K/V： K_full = [K0, K1, K2, K3] V_full = [V0, V1, V2, V3] GPU0: Q0 attend K_full/V_full → H0^(l) GPU1: Q1 attend K_full/V_full → H1^(l) GPU2: Q2 attend K_full/V_full → H2^(l) GPU3: Q3 attend K_full/V_full → H3^(l)
一句话：Q 表示“我负责哪些 token 的输出”，所以 Q 可以留在本地；K/V 表示“上下文里有哪些信息可以被看见”，所以在 Full K/V 策略里每张 GPU 都拿一份完整 K/V。
常见误解纠正：
不是“先算 KV，再聚合 KV，再算 Q，再聚合 Q”。
更准确是：每张 GPU 先同时算本地 Q/K/V，然后只聚合 K/V，Q 不聚合。

7.因果掩码与顺序保证

7.1 每张 GPU 拿到完整 K/V，会不会看到未来？

不会。因为 causal mask 会把未来位置 mask 掉。

假设 4 个 chunk：

K0: token 0 ~ 31 K1: token 32 ~ 63 K2: token 64 ~ 95 K3: token 96 ~ 127

块级可见性是下三角：

K0 K1 K2 K3
Q0 可见 未来 未来 未来
Q1 可见 块内按下三角 mask 未来 未来
Q2 可见 可见 块内按下三角 mask 未来
Q3 可见 可见 可见 块内按下三角 mask
重要：“收到 K/V”不等于“真正使用这些 K/V”。即使某张 GPU 收到了未来块的 K/V，causal mask 也会把对应 attention score 变成负无穷，softmax 后权重为 0。实现里也可以跳过完全未来的块。

7.2 聚合时怎么保证顺序？

顺序由三类信息保证：

  • 固定 token range：GPU0 负责 token 0~32k，GPU1 负责 32k~64k，等等。
  • global position id：每个 token 有全局位置，比如 token32768 的 position id 是 32768。
  • KV block metadata：实际系统会以 block / page 形式管理 KV cache，metadata 记录某块属于哪个请求、哪个位置范围。
K_full = concat(K0, K1, K2, K3) V_full = concat(V0, V1, V2, V3) 或者在 paged/block KV 中： block_id → request_id + token_position_range + gpu_rank

8.Ring Attention

Ring Attention 用于“Partial Query, Partial K/V”这种场景：每张 GPU 只保存自己那块 K/V，不再让每张 GPU 都保存完整 K/V。

一句话：Query 留在本地，K/V 沿着 GPU 环形传递；每张 GPU 每收到一块 K/V，就计算一部分 attention，并用 online softmax 合并，最后得到等价于完整 K/V attention 的结果。

为什么需要 Ring Attention？

如果上下文太长，完整 K/V 单卡放不下，不能再做 Full K/V gather。于是每张 GPU 只存一部分 K/V：

GPU0: Q0, K0, V0 GPU1: Q1, K1, V1 GPU2: Q2, K2, V2 GPU3: Q3, K3, V3

但 attention 又要求每个 query 能看到所有合法历史 K/V。Ring Attention 的做法是让 K/V 分块流动：

GPU0 → GPU1 → GPU2 → GPU3 → GPU0

4 GPU 的通信过程

第 0 轮：每张 GPU 用本地 K/V GPU0: Q0 attend K0/V0 GPU1: Q1 attend K1/V1 GPU2: Q2 attend K2/V2 GPU3: Q3 attend K3/V3 第 1 轮：K/V 转一格 GPU0 收到 K3/V3，计算 Q0 attend K3/V3 GPU1 收到 K0/V0，计算 Q1 attend K0/V0 GPU2 收到 K1/V1，计算 Q2 attend K1/V1 GPU3 收到 K2/V2，计算 Q3 attend K2/V2 第 2 轮：继续转 ... 经过若干轮，每个 Q 都接触过需要的 K/V 块。

注意 causal mask

对 causal LLM 来说，Q1 并不是有效看 K2/K3。它可能在通信中收到 K2/K3，但这些未来块会被 mask 或跳过。

以 Q1 = token 32~63 为例： 有效使用： K0/V0: token 0~31，全可见 K1/V1: token 32~63，块内下三角可见 无效 / 被 mask： K2/V2: token 64~95，未来 K3/V3: token 96~127，未来

为什么不能简单把 partial result 相加？

因为 attention 里有 softmax：

Attention(Q, K, V) = softmax(QK^T / sqrt(d)) V

softmax 的分母需要所有合法 K 的分数共同参与，所以分块计算时要维护：

  • 当前最大 score，保证数值稳定；
  • 当前 softmax 分母；
  • 当前加权 Value 累积结果。

这就是 online softmax / streaming softmax 的思想。

9.DCP：Decode Context Parallel

Decode 阶段每一步只有很少的新 query token，但它要 attend 大量历史 KV。vLLM 文档说，DCP 的核心是如何把 paged KV cache 切分到多张 GPU 上。

9.1 为什么 DCP 重点是 KV cache？

decode 过程中，历史上下文会越来越长：

初始：prompt 128k tokens 生成 1 个 token 后：128k + 1 生成 1000 个 token 后：129k ...

每一层都要保存历史 K/V，所以 KV cache 非常占显存。DCP 把 KV cache 沿 T 维度切开。

9.2 DCP decode 一步怎么生成一个 token？

假设 4 张 GPU，历史 KV 被按 token 维度切开：

GPU0: token 0 ~ 32k 的 KV GPU1: token 32k ~ 64k 的 KV GPU2: token 64k ~ 96k 的 KV GPU3: token 96k ~ 128k 的 KV

当前新 token 在最后，所以它可以看全部历史 KV。某一层中：

当前 hidden state ↓ 算出当前 token 的 Q ↓ Q 被复制 / 分发到 DCP 组 ↓ GPU0: Q attend KV0 → partial attention info GPU1: Q attend KV1 → partial attention info GPU2: Q attend KV2 → partial attention info GPU3: Q attend KV3 → partial attention info ↓ 合并 partial attention info ↓ 得到这一层完整 attention output ↓ 进入 MLP / residual / 下一层
重点：DCP 不是每张卡各自吐一个 token。多张卡共同为同一个新 token 计算 attention；每一层都要合并 partial attention result，所有层完成后经过 LM head 得到 logits，最后才能采样 / 贪心生成下一个 token。

9.3 TP 与 DCP 的关系

普通 Tensor Parallel 通常先沿 KV head 维度切 KV cache；但当 KV heads 很少时，继续增加 TP 会导致 KV cache 在 TP 组内重复。DCP 再沿 token 维度 T 切，减少重复。

并行方式 主要切分维度 主要解决
Tensor Parallel -tp 模型权重 / attention heads / KV heads 模型太大、单卡放不下；并行算矩阵。
Decode Context Parallel -dcp 上下文 token 维度 T KV cache 重复与显存压力。

9.4 DCP size 怎么理解？

DCP size 不增加需要启动的 GPU 数量，它复用 TP group 里的 GPU。vLLM CLI 文档也说明，tp_size 需要能被 dcp_size 整除。

取舍：更大的 dcp_size 可以减少 KV cache duplication，但会增加通信开销。不是越大越好。

9.5 文档里的 case study

模型 文档中的情况 建议理解
DeepSeek-R1 MLA 启用时 1 个 KV head，-tp 8 会有 8 倍 KV cache duplication，可加 -dcp 8。 KV heads 太少，TP 组内 KV 重复严重，DCP 沿 T 维度再切。
Kimi-K2 -tp 16 时 KV cache 可能 16 倍重复，-dcp 16 可完全去重，但通信开销更大；-dcp 8 可折中。 显存节省与通信开销之间取平衡。
Qwen3-235B-A22B 4 个 KV heads，-tp 8 时 duplication 为 2 倍，可加 -dcp 2 去掉重复。 DCP size 可按重复倍数选择。

11.1 Q/K/V 的 PyTorch 示例

下面代码展示 Q/K/V 从 hidden states 线性投影而来。它不是 vLLM 内部实现，只是教学示例。

import torch
import math

# 假设 batch_size = 1, seq_len = 3, hidden_dim = 4
H = torch.tensor([[
    [1.0, 0.0, 1.0, 0.0],  # token0
    [0.0, 1.0, 1.0, 0.0],  # token1
    [0.0, 0.0, 1.0, 1.0],  # token2
]])

hidden_dim = 4
head_dim = 2

# 三组训练好的权重矩阵；这里随机初始化只是示例
torch.manual_seed(0)
W_Q = torch.randn(hidden_dim, head_dim)
W_K = torch.randn(hidden_dim, head_dim)
W_V = torch.randn(hidden_dim, head_dim)

Q = H @ W_Q
K = H @ W_K
V = H @ W_V

print("Q shape:", Q.shape)  # [1, 3, 2]
print("K shape:", K.shape)  # [1, 3, 2]
print("V shape:", V.shape)  # [1, 3, 2]

11.2 带 causal mask 的 attention

def causal_attention(Q, K, V):
    """
    Q, K, V: [batch, seq_len, head_dim]
    返回: attention output [batch, seq_len, head_dim]
    """
    batch, seq_len, head_dim = Q.shape

    scores = Q @ K.transpose(-1, -2) / math.sqrt(head_dim)
    # scores: [batch, seq_len, seq_len]

    # causal mask: 上三角为 True，表示未来位置
    mask = torch.triu(
        torch.ones(seq_len, seq_len, dtype=torch.bool),
        diagonal=1
    )

    scores = scores.masked_fill(mask, float("-inf"))
    weights = torch.softmax(scores, dim=-1)
    output = weights @ V
    return output

O = causal_attention(Q, K, V)
print("output shape:", O.shape)

11.4 Ring Attention 的伪代码

下面是逻辑伪代码，重点看“Q 不动，K/V 转圈”。真实实现会更复杂，并且要用 online softmax 保证和完整 attention 等价。

# 每张 GPU:
# Q_local: 自己负责的 query chunk
# K_cur, V_cur: 当前手里的 K/V chunk
# rank: 当前 GPU 编号
# world_size: GPU 数

state = OnlineSoftmaxState()

for step in range(world_size):
    # 根据当前 Q 和收到的 K/V chunk 计算局部 score 和 value 累积
    # 内部要根据 global position 做 causal mask
    state.update(Q_local, K_cur, V_cur, causal_mask=True)

    # 把当前 K/V 发给下一个 GPU，同时从上一个 GPU 收新的 K/V
    send_to = (rank + 1) % world_size
    recv_from = (rank - 1 + world_size) % world_size
    K_cur, V_cur = send_recv(K_cur, V_cur, send_to, recv_from)

O_local = state.finalize()

11.5 Online softmax 合并思想

分块 attention 不能简单相加，因为 softmax 要全局归一化。下面是单个 query 的简化版 online softmax。

def online_attention_one_query(q, K_blocks, V_blocks):
    """
    教学示例：单个 query 对多个 K/V block 做 streaming attention。
    q: [d]
    K_block: [block_len, d]
    V_block: [block_len, d]
    """
    m = torch.tensor(float("-inf"))  # 全局最大 score
    l = torch.tensor(0.0)            # softmax 分母
    out = torch.zeros_like(q)        # 加权 V 累积

    for K, V in zip(K_blocks, V_blocks):
        scores = (K @ q) / math.sqrt(q.numel())  # [block_len]

        # block 内最大分数
        m_new = torch.maximum(m, scores.max())

        # 旧累积项需要按新的 m_new 重缩放
        old_scale = torch.exp(m - m_new)
        new_weights = torch.exp(scores - m_new)

        out = out * old_scale + new_weights @ V
        l = l * old_scale + new_weights.sum()
        m = m_new

    return out / l

logits = lm_head(hidden)
next_token = sample_or_argmax(logits)

# 把新 token 在每一层产生的 K/V 按 DCP 的 T 维度策略写入 KV cache
append_new_kv_to_sharded_cache(next_token)

12.速查表

问题 答案
Prefill 是什么？ 读取完整 prompt，计算各层 hidden states 与 prompt 的 KV cache，并准备第一个输出 token。
Decode 是什么？ 基于历史 KV cache，自回归地一个 token 一个 token 生成。
Q/K/V 从哪里来？ 由当前层 hidden states 乘以本层的 W_Q/W_K/W_V 得到。
每层会重新算 Q/K/V 吗？ 会。第 l 层使用 H^(l-1) 计算本层的 Q/K/V。
PCP 的 Partial Query, Full K/V 是什么？ 每张 GPU 只负责一块 Query，但通过通信拿到完整 K/V，然后算自己那块 token 的输出。
Q 要聚合吗？ 在 Partial Query, Full K/V 中，Q 不需要聚合；聚合的是 K/V。
拿到完整 K/V 会不会看到未来？ 不会，causal mask 会屏蔽未来位置。
Ring Attention 是什么？ K/V 不一次性聚合，而是在 GPU 环里分块传递；每张 GPU 边收 K/V 边算 attention，并用 online softmax 合并。
DCP 是什么？ Decode 阶段把历史 KV cache 沿 token 维度 T 切到多张 GPU，减少 KV cache duplication。
DCP 会增加 GPU 数吗？ 通常不会；它复用 TP group 内的 GPU。
DCP 中每张卡算完就能吐 token 吗？ 不能。每层都要合并 partial attention result，所有层结束并经过 LM head 后才能生成下一个 token。
最终记忆版：
PCP：长 prompt 太长，大家一起读。
Ring Attention：完整 K/V 放不下，就让 K/V 在 GPU 之间传着看。
DCP：decode 时历史笔记太厚，大家分开存；每次回答前，把相关结果合并后再继续。
